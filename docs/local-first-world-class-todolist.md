## 2. 北极星定义

### 2.1 唯一目标

把 AI Director Studio 做成一台**本地单机运行**、但在内容质量、角色一致性、返工效率、导演控制力上足够接近专业工作室流程的 AI 短剧 / 漫剧生产系统。

### 2.2 目标产物

系统必须能产出两类内容：

- 短剧成片
  - 竖屏优先，默认 `9:16`
  - 也支持 `16:9`
  - 以剧情短剧、漫改短剧、情绪型短剧情境为主
- 漫剧成片
  - 竖版长图优先
  - 支持带对白气泡、旁白框、章节封面、拟声字

### 2.3 成功标准

本项目成功，不以“能调用多少模型”为标准，而以以下结果为标准：

- 角色跨镜头、跨场次、跨剧集保持稳定
- 单集生成过程中废片率显著下降
- 用户可以对失败镜头快速定位、重做、对比、替换
- 每一步产物都能保存、复用、回滚、继续打磨
- 系统逐渐形成角色库、风格库、镜头库、提示词库，而不是每集从零开始

### 2.4 失败定义

以下情况视为方向失败：

- 工具只能“批量调用接口”，不能稳定做出可用成片
- 一次生成后无法精修，只能全量重跑
- 角色、风格、镜头语言大面积漂移
- 长任务中断后无法恢复
- 内容质量完全依赖运气，无法通过结构化流程提升

## 3. 产品定位

### 3.1 这是一个什么产品

这是一个：

- 本地优先工具
- 单用户导演工作台
- 资产驱动的 AI 内容生产系统
- 以“持续打磨”和“高一致性”为核心的影视 / 漫剧制作机

### 3.2 这不是什么产品

这不是：

- SaaS 协作平台
- 面向多人在线编辑的云工具
- 只做文生图 / 图生视频的一键玩具
- 通用视频剪辑器
- 面向直播、营销剪辑、真人实拍管理的系统

### 3.3 用户画像

0-1 阶段假定只有一个核心用户：

- 会写故事大纲
- 懂短剧节奏
- 愿意手动筛片、重做、精修
- 愿意经营角色 IP 和内容风格
- 接受本地长任务、显存 / 算力 / API 成本约束

## 4. 外部项目借鉴原则

以下项目和方向可以借鉴思想，但不盲目照搬：

- `waoowaoo`
- `Jellyfish`
- `Toonflow`
- `Prisma + 本地 FFmpeg` 这一类本地生产管线方案

这里的借鉴重点不是“界面长什么样”，而是“它们为什么在生产流程上成立”。

### 4.0 本次调研的直接结论

结合本次检索到的公开项目信息，可以提炼出几条非常明确的方向：

- `Jellyfish` 的价值不在“接了很多模型”，而在于把短剧生产拆成了
  - 剧本理解
  - 场次 / 镜头准备
  - 资产一致性管理
  - shot 级生成
  - 统一任务中心
- `Toonflow` 的价值不在“桌面端”本身，而在于它证明了
  - 本地 / 轻部署形态适合短剧创作
  - 从剧本到分镜到角色到视频的全链路是用户能理解的
- `Prisma + 本地 FFmpeg` 这一类方案的价值在于
  - 数据可结构化
  - 任务可追踪
  - 媒体可落盘
  - 合成能力不依赖云后期

因此，本项目 0-1 最正确的方向不是“更多 provider”，而是：

- 更强的镜头工作流
- 更强的资产一致性
- 更强的任务恢复与审片能力

### 4.1 应借鉴的共性思想

- 资产优先，而不是任务优先
- 时间线和镜头组织优先，而不是单个 API 优先
- 允许多候选、多版本、多轮筛选
- 强调重做成本低，而不是一步到位
- 把生成、审查、编辑、导出视为一个系统，而不是四个分离模块

### 4.2 不应盲目复制的东西

- 复杂的云端协作设计
- 过早引入账号、权限、支付、在线存储
- 为了“像 SaaS”而牺牲本地生产效率
- 过度抽象导致单机版本反而更难用

### 4.3 启发结论

本项目应当是：

- `本地创意工作站`
- `角色资产管理系统`
- `镜头与时间线导演台`
- `AI 生成编排器`
- `后期组装器`

而不是单纯的“多模型调用面板”。

### 4.4 借鉴对象到设计决策的映射

#### 从 Jellyfish 借鉴什么

- 采用 `chapter / scene / shot / task` 这类更接近生产现场的组织模型
- 把“准备完成”和“开始生成”分开
- 把候选资产、对白候选、镜头就绪状态当作正式工作流状态
- 把任务中心做成统一入口，而不是散落在按钮旁

对本项目的落地决策：

- `Scene` 不能再直接等价于“一个最终镜头结果”
- 必须新增 `Shot` 和 `Take`
- 必须新增“镜头准备完成度”的概念
- 必须新增全局任务中心

#### 从 Toonflow 借鉴什么

- 保持从剧本到成片的一体化体验
- 用面向创作者的流程组织复杂能力，而不是暴露底层接口
- 把角色、分镜、视频这些高频对象做成稳定入口

对本项目的落地决策：

- 项目主页必须是工作台，而不是数据列表页
- 角色库、场次 / 镜头工作台、导出页必须是一等页面
- 所有复杂 AI 能力必须包裹在导演可理解的操作语义里

#### 从 Prisma + 本地 FFmpeg 借鉴什么

- 结构化数据 + 本地媒体落盘是正确组合
- 最终成片能力必须握在本地
- 本地导出、回看、替换、重导是核心能力，不是附属功能

对本项目的落地决策：

- 数据库存“语义和状态”
- 文件系统存“媒体和中间产物”
- FFmpeg 管线必须继续增强，而不是弱化
- 导出记录必须长期保存

#### 对 waoowaoo 的使用原则

- 如果后续确认其在资产编排、导演流或 agent 协作上有可借鉴设计，可继续吸收
- 在没有稳定公开实现细节之前，不把它作为结构性依赖来源

对本项目的落地决策：

- 借鉴思想可以继续扩充
- 结构设计优先基于已验证的本地工作流原则，不依赖单一外部项目命名

## 5. 产品哲学

### 5.1 First-Class Asset

角色、镜头、风格、音色、提示词、候选结果、最终采用结果，全部是资产。

系统必须围绕“资产的创建、继承、筛选、替换、复用、淘汰”设计，而不是围绕“请求发出去没有”设计。

### 5.2 Human-in-the-Loop

本项目默认用户是导演，不是 API 调度器。

系统必须允许人工：

- 改镜头
- 改提示词
- 锁定人物
- 比较候选
- 废弃失败版本
- 手动指定采用结果

### 5.3 Quality Loop over One-shot

高质量内容来自闭环，不来自一次好运。

默认流程必须支持：

- 生成
- 审查
- 打分
- 重做
- 选择
- 固化

### 5.4 Local-First Reliability

既然只在本地跑，就必须尊重本地生产现实：

- 会断网
- 会切换窗口
- 会关机
- 会中断任务
- 会磁盘爆满
- 会 API 限流

所以任务和资产状态必须持久化，不能只存在内存里。

### 5.5 Structured Creativity

创作不是乱输 prompt。

系统必须用结构化信息约束自由生成：

- 角色圣经
- 风格圣经
- 场景结构
- 镜头模板
- 叙事模板
- 音频模板

## 6. 0-1 范围边界

### 6.1 必做

- 单项目多剧集
- 角色资产管理
- 剧本拆解为场次 / 镜头
- 首帧多候选生成
- 视频生成与质检重试
- 角色级配音基础能力
- SFX / BGM 基础能力
- 本地 FFmpeg 合成
- 漫剧导出基础版
- 任务持久化与恢复
- QA 面板
- 版本对比与手动采用

### 6.2 暂不做

- 在线部署
- 多用户协作
- 权限系统
- 云素材库
- 分享链接
- 商业化后台
- 自动发布平台
- 移动端优先适配

### 6.3 有条件后做

- 多人协作
- 云端备份
- 模型成本分析中心
- 更复杂的自动导演 agent
- 自动投放 / 自动分发

## 7. 最终产品应有的心智模型

系统内部必须围绕以下对象建模：

- `Project`
  - 一部作品
- `StyleBible`
  - 一部作品的视觉和叙事规则
- `CharacterBible`
  - 每个角色的固定设定和变化边界
- `Episode`
  - 单集
- `Scene`
  - 场次，承载剧情目的和时空语义
- `Shot`
  - 具体镜头，承载生成任务
- `Take`
  - 某镜头某次候选结果
- `Task`
  - 某个生成 / 质检 / 导出任务
- `Review`
  - 某次质量评审结果
- `Export`
  - 某次成片导出结果

如果后续数据结构无法映射到这些心智对象，说明设计偏离了方向。

## 8. 信息架构

### 8.1 Project 层

每个项目至少包含：

- 基本信息
  - 标题
  - 类型
  - 画幅默认值
  - 目标平台
- 世界观
  - 核心设定
  - 时代 / 地点 / 权力结构
  - 禁改规则
- 风格圣经
  - 视觉基调
  - 镜头偏好
  - 服化道规则
  - 光影规则
  - 负面约束
- 角色库
- 剧集列表
- 输出历史

### 8.2 CharacterBible 层

每个角色不能只存 `name/prompt/refImageUrl`。

必须至少包含：

- 标识信息
  - 名字
  - 别名
  - 性别
  - 年龄段
  - 身份
- 稳定描述
  - 五官
  - 发型
  - 身材
  - 服装主基调
  - 气质关键词
  - 典型表情
  - 典型动作
- 不可变锚点
  - 必须保留的脸部特征
  - 必须保留的发色 / 发型
  - 必须保留的穿搭母题
- 可变范围
  - 可换装范围
  - 可接受情绪极值
  - 可接受场景装束
- 关系信息
  - 与主角 / 配角 / 反派关系
- 声音信息
  - 默认 voice profile
- 资产信息
  - 多张定妆照
  - 多角度照
  - 表情照
  - 服装 variations

### 8.3 StyleBible 层

每个项目必须有风格圣经，至少包含：

- 题材标签
- 视觉流派
- 色彩策略
- 镜头偏好
- 画面密度
- 参考时代气质
- 布景约束
- 道具约束
- 负面词库
- 漫剧模式排版风格

### 8.4 Episode 层

每集必须有：

- 本集主题
- 剧情摘要
- 本集 hook
- 结尾悬念
- 上集承接点
- 场次列表
- 本集导出记录

### 8.5 Scene 层

`Scene` 不是简单“第几个分镜”，而是一个场次。

至少包含：

- 场次编号
- 时空信息
  - 地点
  - 时间
  - 白天 / 夜晚
- 出场人物
- 场次目标
  - 剧情推进
  - 情绪升级
  - 信息揭示
- 场次摘要
- 该场次下的镜头列表

### 8.6 Shot 层

每个 `Shot` 至少包含：

- 镜头编号
- 归属场次
- 景别
- 机位
- 镜头运动
- 时长预期
- 主体角色
- 行为描述
- 叙事目的
- 情绪目标
- 视觉 prompt
- 音频 prompt
- 多候选 take
- 当前采用 take

### 8.7 Take 层

每个 `Take` 表示一版候选结果，必须可追踪：

- provider
- 输入参数快照
- prompt 快照
- 参考资产
- 生成时间
- 生成耗时
- 本地产物路径
- 自动评分
- 人工评分
- 是否采用
- 是否废弃
- 废弃原因

## 9. 目录与本地资产策略

### 9.1 资产目录必须专业化

不再把所有文件平铺进 `public/workspace/`。

推荐目标结构：

```text
public/workspace/
  projects/
    {projectId}/
      project.json
      style-bible.json
      characters/
        {characterId}/
          bible.json
          refs/
          angles/
          expressions/
          wardrobe/
      episodes/
        {episodeId}/
          episode.json
          scenes/
            {sceneId}/
              scene.json
              shots/
                {shotId}/
                  shot.json
                  takes/
                    {takeId}/
                      input.json
                      review.json
                      image.jpg
                      video.mp4
                      audio.mp3
                      sfx.mp3
          exports/
      cache/
      temp/
```

### 9.2 资产管理原则

- 原始候选和最终采用结果必须分离
- 所有生成参数必须可回溯
- 孤儿资产可被扫描和清理
- 临时文件和长期资产分离
- 导出成片必须有 manifest

## 10. 核心工作流

### 10.1 全链路流程

1. 创建项目
2. 建立世界观和风格圣经
3. 建立角色圣经与定妆资产
4. 新建剧集
5. 输入大纲 / 梗概 / 剧本
6. 拆解为场次与镜头
7. 发现新角色则进入角色拦截流程
8. 为每个镜头生成首帧多候选
9. 人工挑选或局部重做首帧
10. 基于采用首帧生成视频候选
11. 对视频进行多轮 QA
12. 生成角色级对白配音
13. 生成环境音和 BGM
14. 进入镜头审片 / 替换 / 修复
15. 按镜头组合成片
16. 导出短剧版
17. 导出漫剧版
18. 固化本集记忆，生成下一集 seed

### 10.2 新角色拦截流程

该流程必须保留，并升级为正式能力：

- 解析剧本时识别新重要角色
- 返回 `NEED_CHARACTER_SETUP`
- 阻断后续分镜入库或生成
- 用户完成角色圣经和首批定妆资产
- 系统恢复场次 / 镜头写入

### 10.3 角色一致性流程

角色一致性不是“用一张 ref 图”就结束。

必须有：

- 定妆阶段
- 多角度补图阶段
- 表情 / 情绪扩展阶段
- 不同服装场景变体阶段
- 后续镜头持续比对阶段

### 10.4 生成闭环流程

每类生成的标准流程都是：

1. 准备输入
2. 选择 provider
3. 生成候选
4. 自动质量检测
5. 打分
6. 人工选择 / 驳回
7. 必要时重试
8. 固化采用结果

## 11. 质量门槛

### 11.1 图像质量门槛

任何首帧候选必须至少通过以下检查：

- 主体角色脸部可辨识
- 手部和肢体无明显崩坏
- 构图满足镜头意图
- 服饰不偏离角色圣经
- 光影和色彩不脱离风格圣经
- 情绪表达与镜头目标匹配

### 11.2 视频质量门槛

任何视频候选必须至少通过以下检查：

- 首帧与采用首帧接近
- 主体身份稳定
- 动作不严重崩坏
- 时长符合预期
- 关键段落不出现闪烁和主体漂移
- 可接受的物理运动逻辑

### 11.3 剧本文本质量门槛

场次 / 镜头拆解必须具备：

- 节奏清晰
- 角色动机合理
- 场次目标明确
- 情绪曲线可读
- 有 hook，不拖沓
- 镜头意图明确，不是泛 prompt

### 11.4 音频质量门槛

- 对白清晰
- 音色与角色匹配
- 情绪正确
- 底噪可控
- 响度可控
- 音画同步基本准确

### 11.5 漫剧质量门槛

- 格子排版可读
- 对白气泡不遮脸
- 页面节奏不乱
- 主信息不被装饰淹没
- 封面与章节标题清晰

## 12. QA 系统设计

### 12.1 自动 QA 不是单点判断

不允许只依赖“抽一帧判断 PASS / FAIL”。

最低要求：

- 图像 QA
- 视频多帧 QA
- 首尾帧一致性 QA
- 人脸一致性 QA
- prompt 合规性 QA
- 音频基础 QA

### 12.2 QA 结果必须结构化

每次 QA 至少记录：

- QA 类型
- 判定结果
- 分数
- 失败原因标签
- 建议动作
  - 直接采用
  - 轻微可接受
  - 必须重做
  - 需要换 provider

### 12.3 QA 面板必须可操作

用户需要能：

- 看见所有失败项
- 按失败类型筛选
- 看见重试次数
- 看见候选比较
- 一键重做
- 指定 provider 重做
- 标记“接受瑕疵”

---

## 13. Prompt 系统设计

### 13.1 Prompt 不能是自由文本黑盒

应拆成结构：

- 风格前缀
- 角色锚点
- 镜头描述
- 场景描述
- 行为描述
- 情绪描述
- 技术质量后缀
- 负面约束

### 13.2 Prompt 资产化要求

- 每次生成保存最终 prompt
- 支持 prompt 模板版本管理
- 支持按角色 / 场景 / provider 维度复用
- 支持 prompt A/B 对比
- 支持“锁定人物只改背景”这类局部重组

### 13.3 反模式

以下方式禁止作为长期方案：

- 用户每次手写整段 prompt
- 只在前端临时改文本，不保留版本
- 生成失败后盲目重试，不修改参数

---

## 14. Provider 策略

### 14.1 Provider 抽象目标

系统必须支持同一能力多 provider 对打，而不是把 provider 绑死在业务流程里。

### 14.2 每类能力的 provider 抽象

- 文本拆解 provider
- 图像生成 provider
- 视频生成 provider
- TTS provider
- SFX provider
- QA / vision judge provider

### 14.3 Provider 评估维度

系统必须持续积累以下统计：

- 成功率
- 平均耗时
- 对角色稳定性的表现
- 对动作镜头的表现
- 对写实风格的表现
- 平均返工次数

### 14.4 Provider 选择策略

每次生成可由系统根据以下维度推荐默认 provider：

- 镜头类型
- 角色稳定要求
- 动作强度
- 写实度要求
- 近期成功率

---

## 15. 音频系统设计

### 15.1 VoiceProfile

每个角色必须能绑定声音画像：

- 声线类型
- 年龄感
- 情绪风格
- 语速
- 停顿偏好
- 默认音量
- 默认语言风格

### 15.2 对白生成要求

- 支持逐句生成
- 支持重音和停顿
- 支持句级重做
- 支持同角色跨镜头声音一致

### 15.3 音效和 BGM

- SFX 不能只是一条混音附件
- 至少区分环境音、动作音、强调音
- BGM 至少支持段落化，而不是全片一首到底

### 15.4 后期最小标准

- 响度标准化
- ducking
- 起止淡入淡出
- 音画对齐微调

---

## 16. 导演工作台设计

### 16.1 必须具备的页面

- 项目总览页
- 角色资产库页
- 剧集总览页
- 场次 / 镜头工作台
- 生成队列页
- QA 审片页
- 导出页

### 16.2 场次 / 镜头工作台必须支持

- 瀑布流看候选
- 时间线看镜头顺序
- 拖拽排序
- 切换 adopted take
- 改 prompt
- 重做指定镜头
- 标记场次问题

### 16.3 审片模式必须支持

- 同镜头多版本并排对比
- 跨集角色一致性抽查
- 快速替换失真镜头
- 标记“待修复”

---

## 17. 任务系统设计

### 17.1 现状问题

内存 Map + SSE 只能演示进度，不能支撑本地生产。

### 17.2 目标任务模型

必须引入持久化任务结构：

- `queued`
- `running`
- `retrying`
- `paused`
- `failed`
- `completed`
- `cancelled`

### 17.3 每个任务必须记录

- 任务类型
- 父任务 / 子任务关系
- 输入对象
- 输出对象
- 尝试次数
- 日志
- 错误原因
- 开始时间 / 结束时间

### 17.4 任务恢复

应用重启后必须能够：

- 识别未完成任务
- 识别部分完成的资产
- 允许从断点恢复
- 避免重复生成已采用资产

## 18. 数据库演进方向

### 18.1 现有 Schema 不够

当前 `Project / Character / Episode / Scene` 是合理起点，但不足以承载世界级本地工作流。

### 18.2 建议新增核心表

- `StyleBible`
- `CharacterBible`
- `Shot`
- `Take`
- `GenerationTask`
- `Review`
- `VoiceProfile`
- `PromptTemplate`
- `ExportRecord`

### 18.3 最重要的演进原则

- 以资产和工作流为中心建模
- 把“候选结果”视为一等公民
- 把“评审结果”持久化
- 让“最终采用结果”明确可追踪

## 19. 漫剧模式不是导出附属品

漫剧必须作为独立产物模式设计，而不是简单把图片和对白拼成长图。

### 19.1 漫剧模式最低要求

- 多种格子模板
- 气泡布局
- 旁白框
- 角色名牌
- 拟声字
- 封面模板
- 章节页模板

### 19.2 漫剧模式高级方向

- 自动避脸布局
- 长对白自动拆页
- 强情绪镜头放大排版
- 爆点页模板

## 23. 分阶段开发路线

### 阶段 A：从原型升级为“可控生产骨架”

目标：

- 数据模型升级
- 任务可恢复
- 角色 / 风格规则正式入库

完成标志：

- 能持久化管理角色圣经和风格圣经
- 能按 `Scene -> Shot -> Take` 工作
- 应用重启后任务和资产状态不丢

验收口径：

- 用户新建项目后，能完整填写并保存风格圣经
- 每个角色能保存多张参考资产和结构化设定
- 剧本拆解写入的是场次和镜头，而不是只有平铺分镜
- 任何生成任务重启应用后都能在数据库看到明确状态

### 阶段 B：从“可控生产骨架”升级为“导演工作台”

目标：

- 多候选筛选
- 审片
- 替换
- 批量重做

完成标志：

- 用户可在镜头层做选择，而不是只能等整集跑完
- 用户可以快速定位失败镜头并修复

验收口径：

- 用户可以为同一镜头查看多个 take
- 用户可以切换 adopted take
- 用户可以批量重做失败镜头
- 用户可以在 QA 面板筛选失败原因并触发修复动作

### 阶段 C：从“导演工作台”升级为“质量飞轮”

目标：

- QA 数据积累
- provider 对比
- prompt 模板化
- 角色长期稳定性提升

完成标志：

- 同类镜头复用率提升
- 废片率下降
- 同角色跨集稳定性显著提升

验收口径：

- 系统能统计 provider 成功率和返工率
- 系统能追踪角色跨镜头稳定性问题
- prompt 模板和风格模板可以被后续项目复用
- 用户能看到哪些策略真正降低了废片率

### 阶段 D：从“质量飞轮”升级为“双模态产线”

目标：

- 短剧和漫剧同时成为正式产物

完成标志：

- 单集可输出短剧版和漫剧版
- 两条产线共享同一套角色和场次资产

验收口径：

- 同一集既可导出短剧视频，也可导出漫剧长图
- 漫剧不是简单拼图，而是有模板、排版和文本系统
- 两类导出都可追踪到相同的场次、镜头、采用 take

这是建议立即进入开发的第一批任务。

### 24.2 路由 / 服务层

- [ ] 拆分项目页 orchestration
- [ ] 新增 `services/` 或 `lib/workflows/`
- [ ] 把“脚本拆解 -> 新角色拦截 -> 场次镜头落库”整理成独立 workflow
- [ ] 把“首帧 -> 视频 -> 音频 -> QA -> 导出”整理成独立 workflow

### 24.3 UI

- [ ] 角色圣经编辑器
- [ ] 风格圣经编辑器
- [ ] 镜头工作台初版
- [ ] QA 面板初版
- [ ] 任务中心初版

### 24.4 基础质量

- [ ] 让 `pnpm lint` 通过
- [ ] 修复核心类型债务
- [ ] 补最小测试
- [ ] 修 build warning

### 24.5 第一阶段的执行顺序

不允许并行乱做，建议严格按以下顺序推进：

1. 先补数据模型
2. 再补任务系统
3. 再补 workflow 服务层
4. 再补角色圣经和风格圣经 UI
5. 再补镜头工作台
6. 最后补 QA 面板和批量操作

原因：

- 数据模型不稳，后面所有 UI 都会返工
- 任务系统不稳，长流程能力都不可信
- workflow 不抽离，页面会继续膨胀
- 没有角色 / 风格规则，镜头工作台也缺少真正的约束来源

### 24.6 第一阶段 Definition of Done

只有同时满足以下条件，阶段 A 才算完成：

- `Project / StyleBible / CharacterBible / Episode / Scene / Shot / Take / Task` 的核心数据结构已落地
- 首帧生成支持多候选和 adopted take
- 视频生成结果有结构化 QA 结果
- 应用关闭后重开，任务和资产状态仍可恢复
- 用户可以手动替换采用的镜头版本
- README、架构文档、本文档三者不再互相冲突

## 25. 最重要的开发原则

### 25.1 先把数据结构和工作流做对，再追求花哨 UI

### 25.2 先让失败可见、可修、可恢复，再追求全自动

### 25.3 先把角色一致性和镜头可控性做强，再追求更多 provider

### 25.4 任何“只为了演示看起来快”的设计，只要损害长期资产积累，都应该拒绝

### 25.5 任何不能解释为“如何帮助做出更好成片”的功能，都不应该进入 P0

## 26. 供 AI 开发者直接遵循的执行规则

当 AI 参与后续开发时，必须默认遵循以下规则：

- 优先补数据模型、状态机、工作流，不优先补装饰性 UI
- 所有新功能都要说明它对应哪一个资产对象
- 所有生成相关功能都要说明输入、输出、任务状态、失败恢复方式
- 所有新页面都要说明它属于导演工作台的哪一部分
- 所有文档更新都要区分“现状”和“目标”
- 任何新增字段都要解释它如何帮助提升一致性、可控性或复用性

## 27. 一句话结论

AI Director Studio 下一阶段最重要的，不是“再接几个模型”，而是把项目从：

**本地 AI 生成原型**

升级为：

**本地角色资产驱动、镜头可控、任务可恢复、质量可闭环的 AI 短剧 / 漫剧制作系统**

这就是 0-1 阶段唯一正确的大方向。
